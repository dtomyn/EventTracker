# FR-010 Topic Clustering and Mind Map

- Category: Functional
- Status: Implemented
- Scope: AI-driven tag assignment per entry and visualization of a tag-based cluster graph (mind map) mapping relationships between topics for the current timeline group.
- Primary Sources: `app/main.py`, `app/services/topics.py`, `app/services/entries.py`, `app/templates/topic_graph.html`, `scripts/compute_topic_clusters.py`

## Requirement Statements

### Tag Generation
- FR-010-01 The application shall use an AI provider to generate multiple topic tags (2 to 5 per entry) for each entry in a timeline group when the compute script is run.
- FR-010-02 Tags shall be generated from the entry title and the first 600 characters of the entry's final text.
- FR-010-03 The AI shall return tags as a JSON array of concise strings (1 to 3 words each), parsed and validated before being applied to entries.
- FR-010-04 The compute script shall merge AI-generated tags into the entry's existing tag set using the existing tag infrastructure (`tags` and `entry_tags` tables), without removing any tags the user has already assigned.
- FR-010-05 If a generated tag already exists on an entry (case-insensitive comparison), it shall not be added again.
- FR-010-06 If the AI provider is unavailable or tag generation fails for a specific entry, that entry shall be skipped gracefully and processing shall continue for the remaining entries.

### Data and Graph Construction
- FR-010-07 The application shall produce a structured JSON payload describing topic graph nodes (one per unique tag) and relationship edges (co-occurrence between tags on the same entry).
- FR-010-08 Each graph node shall represent a single tag and shall be sized relative to the number of entries that carry that tag.
- FR-010-09 Edges shall connect tags that co-occur on the same entry. Edge weight shall reflect the Jaccard-like co-occurrence ratio between the two tags.
- FR-010-10 The application shall cache the generated graph JSON in the `topic_cluster_cache` table, keyed by group id, so the visualization route can serve it synchronously without recomputing.
- FR-010-11 The application shall gracefully return an empty graph when no entries or no tags exist for a group.

### Visualization
- FR-010-12 The application shall provide a dedicated route (`/groups/{group_id}/topics/graph`) to view the topic cluster graph.
- FR-010-13 The application shall expose an API endpoint (`/api/groups/{group_id}/topics`) that returns the cached topic graph as JSON for the client-side renderer.
- FR-010-14 The application shall render the cluster graph on the client side using the `d3.js` library fed by the backend's JSON payload.
- FR-010-15 The visualization shall map tags to graph nodes, scaling the visual size of each node relative to the number of entries carrying that tag.
- FR-010-16 The visualization shall render edges between nodes to represent tag co-occurrence, using edge thickness to indicate co-occurrence strength.
- FR-010-17 The graph implementation shall support dark mode by respecting the host page's `[data-bs-theme="dark"]` CSS custom properties for node, edge, and background colors.

### Scoping and Interactivity
- FR-010-18 The graph data generation shall be strictly bounded to the entries and tags belonging to the currently selected timeline group.
- FR-010-19 The application shall allow the user to click on a topic node in the cluster graph.
- FR-010-20 Clicking a topic node shall redirect the user to a filtered timeline view scoped to entries that carry that tag (via the `q` query string with the tag label).
- FR-010-21 The topic graph view shall provide dedicated zoom in and zoom out controls so the user can adjust the displayed visualization without relying solely on pointer or trackpad gestures.

## Acceptance Notes

- Topic tags are generated by running `uv run python -m scripts.compute_topic_clusters` before viewing the graph. The graph is served from the cache populated by that script.
- The compute script iterates over all timeline groups, generates tags for each entry, merges them into the database, and stores the resulting tag graph in the cache.
- The tag graph is accessible from the main navigation or group sidebar when a valid group is selected.
- The graph gracefully handles small event groups (low entry count) and large event groups without visual crowding or breaking the layout.
- The interactive visualization correctly switches styling when toggling the system or app-level dark mode preference.
- The graph view exposes visible zoom controls that work consistently on desktop and touch-constrained devices.
- Entries without AI-generated tags (e.g. on first run before the script has been executed) still appear in the graph if they carry user-assigned tags.
